System architecture.

Three-tier monorepo: Next.js frontend, Express API service, MongoDB backend. The API is the only thing that talks to Mongo and the provider APIs — the browser never holds an integration token after M6 (encrypted at rest, decrypted in-process for outbound calls). The frontend is a multi-hub shell — Dev (Goal Intelligence) and Manager ship today, with QA and Admin alongside — and the API is engagement-aware, serving the espace (eSpace Server) and crealogix (Cloud) tenancies from one codebase.

02 / Tiers, in one paragraph each

Client · apps/web

Next.js 16 App Router. Renders the shell + every route, calls the API for data. Cookies are signed by the API and travel same-site strict; httpOnly over the same origin (Next rewrites /api/v1/* to localhost:4000 in dev, to the deployed API host in prod). The browser never sees integration tokens — all GitHub/GitLab/Jira calls flow through /api/v1/integrations/proxy/<provider>/* on the API, which decrypts in-process and streams the upstream response back. State management is server-authoritative: goals, specs, context, inputs, snapshots, and tier verdicts persist through the API and read back per session — goal locks are the last localStorage-only holdout (see BL-011). The API is the source of truth.

Service · apps/api

Express + TypeScript. Hosts auth, the encrypted-token proxy, the domain CRUD (goals, snapshots, grading), and the AI relay (chat/grade-pr/classify-goals streams as NDJSON). One folder per route group under src/modules/; each has its own Zod schemas, controller, and router. Auth is cookie-based (M2.3), TOTP enrolment is gated client-side AND server-side (M-CAP follow-up), rate limits are per-IP. Boot pipeline: load env → connect Mongo → apply collection validators + indexes → run migrations → start listener.

Data · MongoDB

One DB (devhub-dev in dev, managed Atlas cluster in prod), 13 collections. Every collection has a $jsonSchema validator (defense-in-depth against malformed writes) and a typed accessor in db/collections.ts. Indexes are ensureIndex'd at boot. Migrations live under db/migrations/ and run after validators — currently m-cap-roles (legacy role → multi-role roles[]) and totp-enrolled-sessions (backfill server-side enforcement flag).

Shared · packages/shared

Cross-workspace primitives. Three subpaths: capabilities/ (M-CAP capability tokens + role-to-cap map), hubs/ (registry of dev/qa/admin/manager hubs + resolution helpers), goal-specs/ (AI-classifier output types + validators). Imported by both web and api via @espace-devhub/shared. Pure JS with ambient .d.ts — no build step, no tests yet.

03 / Key data flows

F1 · Login (with optional TOTP)

  1. Browser POST /api/v1/auth/login with email + password.
  2. API runs argon2id verify (always, even on unknown email — uniform timing).
  3. If user has TOTP enrolled → mint session with totpVerified: false + return needsTotp: true. Frontend swaps to TOTP step; POST /totp/verify flips the flag.
  4. If no TOTP → mint session with totpVerified: true AND totpEnrolled: false. The new server-side gate (requireTotpEnrolled) refuses every protected route until enrolment completes at /totp-setup.
  5. Cookie set httpOnly · same-site=strict · signed; AuthGuard reads /auth/me and routes onward (totp-setup → onboarding → hub).

F2 · Integration proxy (browser → GitHub/GitLab/Jira)

  1. Browser fetch("/api/v1/integrations/proxy/github/...", {credentials: "include"}).
  2. API resolves the session, looks up integrations row keyed on (orgId, userId, providerId).
  3. Token bytes are envelope-decrypted in-process via crypto-secret.ts (AES-256-GCM, key derived from INTEGRATION_TOKEN_KEY). Plaintext stays inside the API process.
  4. Outbound HTTPS to the provider. Response body is piped through to the client via Readable.fromWeb() — no buffering.
  5. Response headers are allowlist-filtered (forward content-type, pagination, rate-limit; drop content-encoding because undici decompresses upstream gzip transparently).
  6. Best-effort lastUsedAt / lastErrorAt update on the integrations row; never blocks the response.

F3 · Snapshots (mirror-mode write + pull-on-session)

  1. User saves a snapshot in the browser → saveSnapshot() writes localStorage, then mirrorSaveSnapshot() POSTs the row.
  2. API enforces the manual-wins-over-auto rule (an auto save can't overwrite an existing manual one for the same week).
  3. On session establishment, <SnapshotsSync /> calls GET /api/v1/snapshots and merges into localStorage via applyPulledSnapshot() — the no-mirror variant (otherwise pull → write → mirror creates a feedback loop).
  4. Auto-snapshotter (useAutoSnapshot) captures one row per completed Sun → Thu work-week on every page visit; idempotent.
  5. Backfill engine (useBackfill) walks every completed week of the current year and synthesises any missing rows from loaded integration data; weeks older than ~90d carry partial: true, gaps: ["events"] because GitHub's events feed caps there.

F4 · Admin edits a user

  1. Admin opens /admin/users, expands a row → AdminUsers fetches via GET /admin/users.
  2. Edits roles / status / hubs / displayName → PATCH /admin/users/:id with minimal patch (only changed fields).
  3. Server validates: every hub in the registry, primaryHub member of allowedHubs, no self-lockout (can't remove own admin or disable self).
  4. If diff is non-empty, write doc + audit row (user.update with field-scoped before/after).
  5. "Reset TOTP" goes through POST /admin/users/:id/totp/reset — clears totpSecret + totpEnrolledAt on the user AND flips totpEnrolled: false on every live session of that user so other tabs immediately bounce to /totp-setup.

F5 · Hub resolution (multi-role users)

  1. Frontend HubsFetcher calls GET /api/v1/hubs/me once per session.
  2. API reads the user's roles[], runs effectiveCapabilities() → Set of capability tokens.
  3. resolveHubsForCapabilities() in @espace-devhub/shared/hubs returns every hub whose requires[] is fully satisfied. Preserves HUB_ORDER.
  4. Per-org overrides from hub_configs merge on top (replace integrations array, partial-merge pages map, hide disabled hubs).
  5. Frontend caches {hubs, primaryHubId, defaultHubId} in hubsStore; the URL slug at /[hub]/* must be in hubs or the HubProvider redirects.

F6 · Manager grades a report's goal

  1. Manager opens a report's board → GET /manager/reports/:userId/goal-health, every read boundary-scoped to managerId === session.userId and capability-gated by manager.team.view.
  2. The grading drawer loads full read-only detail (spec, evidence, AI verdict) via GET /manager/reports/:userId/goals/:goalId/detail.
  3. PUT .../verdict upserts a manager_goal_verdicts row — a durable tier that OUTRANKS the AI goal_tier_verdicts cache wherever a tier is shown.
  4. A best-effort notifications row lands in the engineer's in-app inbox (bell). Delegated-goal judging and BYO-tracker approval reuse the same boundary + notification path.
