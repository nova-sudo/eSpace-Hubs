Deployment guide.

How the stack runs locally today and the recommended shape for production. The app is a monorepo with three deployable units: web (Next.js), api (Express), and mongo. There is no CI/CD pipeline yet — ship via git push + gh pr merge.

01 / Local development

Prerequisites

  • Node 22+ (we use Node 22.21 locally; tsx watch + Next.js 16 want recent runtimes)
  • npm 10+ (workspaces support)
  • Docker Desktop for the Mongo container (or a local Mongo install at :27017)
  • Git Bash on Windows (the team has been running there) — Linux/macOS shells also fine

First-time setup

# 1. Install deps for every workspace
npm install

# 2. Configure env
cp apps/api/.env.example apps/api/.env.local
cp apps/web/.env.example apps/web/.env.local
# Then edit both files — see the env reference below.

# 3. Start the local Mongo (and Redis, reserved for future)
docker compose up -d

# 4. Boot the API + web in two terminals
npm run dev:api   # :4000 — Express + tsx watch
npm run dev:web   # :3000 — Next.js + Turbopack

Open http://localhost:3000. First time, an idempotent seed runs at API boot to create the default org. Use the admin-bootstrap CLI (M2.3b) to create the first admin user, or invite yourself from another admin's session.

Secret hygiene: .env.local files are git-ignored. In production, load env from a secret manager (Doppler, AWS Secrets Manager, Vault, Fly secrets, Render dashboard) — never check anything sensitive into the repo. The two crypto values (SESSION_SECRET + INTEGRATION_TOKEN_KEY) are the most damaging if leaked: compromise of the first lets an attacker forge cookies; compromise of the second lets them decrypt every integration token at rest.

03 / Production deployment shape

Currently none in production. The plan below is what to set up when you ship.

Frontend (Next.js)

Recommended: Vercel or Fly.io.

  • next build → next start on the chosen host
  • Environment: set NEXT_PUBLIC_* + GITHUB_CLIENT_SECRET via the platform's dashboard
  • Next.js rewrites /api/v1/* to the API host server-side — same-origin cookies for free
  • No long-running background work; static-asset CDN handled by the platform

API (Express + TS)

Recommended: Render, Fly.io, or a small ECS task.

  • Build: npm run build:api → run node apps/api/dist/server.js
  • Single process is fine for the current scale (in-process rate-limit store)
  • Scale-out later requires Redis for cross-instance rate limits and session lookups (lookups already hit Mongo, so HA is mainly about the limiter)
  • Healthcheck: GET /healthz — pings Mongo, returns 200 / 503

Database (Mongo)

Recommended: MongoDB Atlas (M10+ shared cluster for early prod).

  • SCRAM auth — the connection string includes the user/password
  • $jsonSchema validators apply on first boot via db/collections.ts
  • Indexes ensured at boot, including TTL on sessions.expiresAt and auth_tokens.expiresAt
  • Migrations (m-cap-roles, totp-enrolled-sessions) run idempotently on every boot

Email (Resend)

For real delivery: set RESEND_DOMAIN_MODE=verified and verify a sending domain.

  • Add the DNS records Resend shows in their dashboard (DKIM, SPF, return-path)
  • Set RESEND_FROM_EMAIL to no-reply@yourdomain.com
  • Sandbox mode is OK for staging — it only delivers to allowlisted addresses

04 / Production checklist

Run through this before flipping DNS to a new deployment.

Secrets

  • SESSION_SECRET is 32-byte random hex (not the dev placeholder)
  • INTEGRATION_TOKEN_KEY is 32-byte random hex (not the dev placeholder) — and rotation procedure is documented
  • Both stored in the secret manager, NOT in the repo or in CI plaintext logs
  • GITHUB_CLIENT_SECRET set; NEXT_PUBLIC_GITHUB_CLIENT_ID matches the OAuth app's prod redirect URI
  • RESEND_API_KEY set; sending domain DNS-verified

Mongo

  • Connection string uses retryWrites=true and a per-env DB name
  • Network allowlist restricts the cluster to the API host's egress IPs
  • Backups configured (Atlas snapshot policy or equivalent)
  • On first boot, you see [db] bootstrap complete + the migration steps complete cleanly in the log

Auth / TOTP

  • NEXT_PUBLIC_AUTH_REQUIRED=true on the web
  • Verify a fresh user gets bounced through /totp-setup after first login
  • Verify direct API access without TOTP-enrol returns 401 totp_enrolment_required
  • Verify a password-reset link from the inbox actually lands on /password-reset?token=… at the prod origin

Integrations

  • Connect a GitHub account end-to-end; verify the proxy decompresses upstream gzip cleanly (no content-encoding echo)
  • Connect a Jira account; verify /search/jql works (NOT the deprecated /search)
  • Confirm tokens in integrations collection are envelope-encrypted (v1.iv.tag.ct shape), not plaintext

Observability

  • API logs ship to your log aggregator (pino JSON → Datadog/Logflare/whatever)
  • Frontend errors captured (Sentry or equivalent) — not yet wired in code; add a Sentry init in app/layout.jsx at deploy time
  • Mongo metrics monitored (Atlas dashboards or self-hosted Prom)
  • Healthcheck endpoint hit by the platform every ≤ 60s

05 / Database migrations

Migrations live under apps/api/src/db/migrations/. The boot pipeline (db/collections.ts → runMigrations()) imports each one after applying validators + ensuring indexes, then awaits its completion before the HTTP listener accepts traffic. Each migration is idempotent and non-fatal on failure — a botched migration doesn't crash the server; it logs and retries on next boot.

Live migrations today

FileWhat it does
m-cap-roles.ts For every user row missing roles[], populates it from the legacy role field (members → ["dev"]; everyone else → [role]).
totp-enrolled-sessions.ts Backfills session.totpEnrolled from the owning user's totpEnrolledAt. Only touches sessions where the field is missing (not null — null is a legitimate state set by admin TOTP reset).

Adding a new migration

# 1. New file under apps/api/src/db/migrations/
touch apps/api/src/db/migrations/your-migration.ts

# 2. Export an idempotent function
export async function runYourMigration(col: Collection<X>) {
  // match only rows that haven't been touched yet
  for await (const row of col.find({ yourField: { $exists: false } })) {
    await col.updateOne({ _id: row._id }, { $set: { yourField: derive(row) } });
  }
}

# 3. Wire it in runMigrations() inside db/collections.ts
try {
  const { runYourMigration } = await import("./migrations/your-migration.js");
  const col = await getYourCollection();
  await runYourMigration(col);
} catch (err) { logger.warn(...) }

# 4. tsc --noEmit and dev-restart to verify the log line
Schema validator changes are trickier than data migrations. Mongo's $jsonSchema validates writes against the CURRENT validator, but never replays past writes. So adding a required field breaks every previous-shape doc on the next write. Pattern: add the field as optional first (data migration backfills it on every row), THEN flip it to required in a follow-up validator change once the migration has been observed to complete in prod.

06 / CI / CD

There is no automated pipeline today. The current workflow is:

  1. Work on a feature branch off main
  2. git push -u origin <branch>
  3. gh pr create with a self-contained description
  4. Verify locally (npm run typecheck, npm run build:web, manual smoke test)
  5. gh pr merge --merge --delete-branch

What's worth adding when you scale up:

  • A GitHub Actions workflow on PR open: npm install → npm run typecheck → npm run build:web + build:api. Catches broken PRs before merge.
  • A deploy-prod.yml on push to main that triggers your host's deploy hook (Vercel auto-deploys today on push; Render needs a hook).
  • Branch protection on main requiring the CI to pass before merge.

07 / Backup & recovery

  • Mongo: Atlas continuous backup or daily snapshot policy on the cluster. PITR (point-in-time recovery) for paid tiers.
  • Encrypted integration tokens: the ciphertext is in Mongo; the key (INTEGRATION_TOKEN_KEY) is in your secret manager. Losing the key without rotation means every token row becomes garbage — users have to reconnect every integration. Document key rotation; consider KMS-managed envelope encryption (per-record DEK wrapped by a KMS-resident KEK) as a follow-up.
  • User-facing recovery: password reset via email (working). TOTP recovery via admin reset (working). Backup codes still missing — when a user loses their phone today, they're locked out until an admin runs the reset, which is a manual process.
  • Sessions: stored in Mongo with TTL; eviction is automatic. Forced-logout-everywhere is one Mongo deleteMany({userId}) away (already used by password-reset).
